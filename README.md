# 1. Что это за проект

**VITO** — это упрощенная копия Авито.

Простыми словами:
- это сайт, где можно смотреть объявления;
- пользователь может зарегистрироваться, войти, создать свое объявление, редактировать его и удалить;
- есть AI-помощь для текста, цены и анализа качества объявления.

Кому это нужно:
- пользователю — чтобы работать с объявлениями;
- разработчику — как учебный/практический проект со связкой frontend + backend + AI.

Почему “похоже на Авито”:
- есть каталог;
- есть карточка объявления;
- есть личный раздел “мои объявления”;
- есть создание/редактирование.

Основная идея проекта:
- дать рабочий базовый маркетплейс-флоу без сложной “энтерпрайз” инфраструктуры.

---

# 2. Из чего состоит проект

Проект состоит из 2 частей.

## Frontend (`/frontend`)

Это клиентская часть (то, что видит пользователь в браузере):
- страницы;
- формы;
- кнопки;
- маршруты;
- логика авторизации на клиенте;
- работа с API backend.

## Backend (`/server`)

Это серверная часть:
- хранит данные объявлений и пользователей (сейчас в памяти процесса);
- отдает API;
- проверяет авторизацию и права доступа;
- проксирует AI-запросы в Ollama.

## База данных

Сейчас отдельной базы данных **нет**.
Данные живут в памяти процесса backend и сбрасываются при перезапуске.

## AI / Ollama

- frontend вызывает backend-ручки `/ai/*`;
- backend вызывает Ollama;
- backend умеет попробовать запустить Ollama автоматически при старте.

## Инфраструктурная часть

В репозитории сейчас **нет**:
- Docker-файлов;
- CI/CD конфигов проекта в корне (типа `.github/workflows` для самого проекта);
- корневых `.env` файлов.

# 4. Что пока не готово или сделано частично

Важно знать заранее:

- отдельной базы данных нет (данные не постоянные);
- refresh token flow не реализован;
- отдельного backend logout endpoint нет (logout делается на frontend);
- API по объявлениям смешанный:
  - чтение/детали/редактирование через `/items`;
  - создание/удаление через `/ads`;
- во frontend есть временный fallback в логике владельца объявления (`canManageAd`) на случай неполных owner-метаданных;
- для create-страницы нет draft в `localStorage` (для edit draft есть).

# 6. Как запустить проект локально

## Что нужно установить заранее

- Node.js (рекомендуется современная LTS-версия);
- npm;
- Ollama (если хотите, чтобы AI-функции точно работали).

## Шаг 1. Запустить backend

```bash
cd server
npm install
npm start
```

Что будет:
- backend поднимется на `http://localhost:8080`;
- попробует автоматически проверить/запустить Ollama.

## Шаг 2. Запустить frontend (в другом терминале)

```bash
cd frontend
npm install
npm run dev
```

Что будет:
- frontend обычно поднимается на `http://localhost:5173`.

## В каком порядке лучше запускать

1. Сначала backend.
2. Потом frontend.

## Как понять, что всё работает

- frontend открывается в браузере;
- `/ads` загружает список;
- логин/регистрация работают;
- AI-кнопки не падают с ошибкой.

# 7. Как части проекта общаются между собой

Простая схема:

1. Пользователь кликает во frontend.
2. Frontend отправляет HTTP-запрос в backend (`http://localhost:8080`).
3. Backend возвращает данные.
4. Для AI:
   - frontend -> backend `/ai/*`;
   - backend -> Ollama;
   - backend -> frontend с результатом.

Важная зависимость:
- если backend не запущен, frontend не сможет работать с данными;
- если Ollama недоступна, обычные страницы работают, но AI-функции дают ошибку.

# 8. Основные пользовательские сценарии

## Гость (без входа)

- открывает `/ads`;
- смотрит список объявлений;
- ищет и фильтрует;
- открывает `/ads/:id`.

## Авторизованный пользователь

- логинится/регистрируется;
- попадает в `/my-ads`;
- создает объявление (`/my-ads/create`);
- редактирует свое объявление;
- удаляет свое объявление;
- использует AI-помощь при создании/редактировании.

## Создание объявления

- пользователь заполняет форму;
- может получить AI-подсказки;
- отправляет форму;
- после успеха открывается страница созданного объявления.

## Редактирование объявления

- доступно только владельцу;
- если объявление чужое — доступ блокируется;
- можно сохранить изменения или удалить объявление.

# 11. Ограничения и спорные места

Что может запутать нового человека:

- смешение путей `/items` и `/ads` для одной сущности объявления;
- отсутствие постоянного хранилища (данные теряются после рестарта backend);
- токены тоже живут в памяти backend и после рестарта становятся невалидными;
- AI во frontend идет через backend `/ai/*`, а не напрямую в Ollama;
- во frontend есть временный ownership fallback (`assumeOwnOnMyAdsRoute`);
- нет отдельной infra-обвязки (docker/ci/cd) в текущем репозитории.

Про `ownerId / isMine`:
- backend сейчас уже отдает эти поля в ключевых ручках;
- но во frontend все еще оставлен fallback на случай неполных данных.

Про delete:
- удаление сейчас реализовано (`DELETE /ads/:id`) и используется в UI.

# 12. Короткая памятка для нового человека

Если ты только открыл VITO, запомни вот это:

1. В проекте две части: `frontend` и `server`, запускать их нужно отдельно.
2. Backend работает на `8080`, frontend обычно на `5173`.
3. Без запущенного backend frontend не сможет показывать реальные данные.
4. Данные backend пока не постоянные: перезапустил сервер — данные сбросились.
5. Для AI frontend ходит в backend `/ai/*`, а backend уже ходит в Ollama.
6. Приватные страницы (`/my-ads*`, edit/create) требуют авторизацию.
7. Токен хранится в `localStorage` на frontend.
8. API по объявлениям сейчас смешанный: `/items` + `/ads`.
9. Подробности по frontend и backend лежат в `frontend/README.md` и `server/README.md`.
