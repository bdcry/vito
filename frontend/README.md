# 1. Что это за frontend

Это клиентская часть проекта **VITO** (упрощенная копия Авито).

Простыми словами:
- backend — это сервер с данными;
- frontend — это сайт, который видит пользователь;
- frontend показывает страницы, формы, кнопки и отправляет запросы в backend.

Что пользователь может делать в приложении:
- смотреть все объявления;
- фильтровать и искать объявления;
- открывать карточку объявления;
- регистрироваться и входить;
- смотреть свои объявления;
- создавать, редактировать и удалять свои объявления;
- пользоваться AI-подсказками при создании/редактировании.

Зачем нужен проект:
- это учебный/практический проект с базовым маркетплейс-флоу (каталог + личный раздел + авторизация + AI-помощь).

---

# 2. Что уже работает

На текущий момент во frontend реально работает:

- публичная страница всех объявлений (`/ads`);
- поиск, сортировка, фильтры, пагинация;
- публичная деталка объявления (`/ads/:id`);
- блок продавца на деталке (имя + email + заглушка-аватар);
- личный раздел “Мои объявления” (`/my-ads`);
- создание объявления (`/my-ads/create`);
- редактирование объявления (`/my-ads/:id/edit` и `/ads/:id/edit` с проверкой прав);
- удаление объявления (из деталки и из редактирования, где это разрешено);
- регистрация и логин;
- хранение токена и восстановление сессии после перезагрузки;
- logout;
- AI-помощники:
  - генерация/улучшение описания;
  - подсказка рыночной цены;
  - AI-анализ объявления (итог, сильные, слабые, рекомендации).

# 3. Как запустить frontend

## Шаг 1. Перейти в папку frontend

```bash
cd frontend
```

## Шаг 2. Установить зависимости

```bash
npm install
```

## Шаг 3. Запустить dev-сервер

```bash
npm run dev
```

После запуска сайт обычно открывается по адресу:

- `http://localhost:5173`

## Что должно быть запущено рядом

Обязательно:
- backend VITO на `http://localhost:8080` (иначе данные не загрузятся).

Для AI-функций:
- frontend ходит в backend (`/ai/*`), а уже backend работает с Ollama.
- то есть frontend напрямую Ollama не вызывает.

# 4. Как устроены страницы приложения

Ниже все реальные роуты из кода.

## `/`

- Что делает: сразу перенаправляет на `/ads`.
- Доступ: публичный.

## `/ads`

- Что делает: показывает общий каталог объявлений всех пользователей.
- Доступ: публичный.

## `/ads/:id`

- Что делает: показывает детали конкретного объявления.
- Доступ: публичный.

## `/ads/:id/edit`

- Что делает: страница редактирования объявления.
- Доступ: защищенный (только авторизованный пользователь).
- Дополнительно: даже если пользователь авторизован, редактировать можно только свое объявление. Для чужого объявления показывается “Нет доступа”.

## `/my-ads`

- Что делает: личный список объявлений текущего пользователя.
- Доступ: защищенный.
- Кто может попасть: только авторизованный пользователь.

## `/my-ads/create`

- Что делает: создание нового объявления.
- Доступ: защищенный.
- Кто может попасть: только авторизованный пользователь.

## `/my-ads/:id`

- Что делает: деталка объявления внутри личного раздела.
- Доступ: защищенный.
- Кто может попасть: только авторизованный пользователь.

## `/my-ads/:id/edit`

- Что делает: редактирование объявления из личного раздела.
- Доступ: защищенный.
- Кто может попасть: только авторизованный пользователь + только владелец объявления.

## `/login`

- Что делает: вход в аккаунт.
- Доступ: публичный.
- Особенность: если уже залогинен, страница сразу отправит на `/my-ads`.

## `/register`

- Что делает: регистрация нового аккаунта.
- Доступ: публичный.
- Особенность: если уже залогинен, страница сразу отправит на `/my-ads`.

## `*` (любой неизвестный путь)

- Что делает: перенаправляет на `/ads`.
- Доступ: публичный.

# 5. Как работает авторизация

Объяснение простыми словами.

## Что делает login

- отправляет email и пароль на backend (`POST /auth/login`);
- получает `accessToken` и профиль пользователя;
- сохраняет токен в `localStorage`;
- обновляет состояние авторизации в приложении.

## Что делает register

- отправляет `fullName`, `email`, `password` на backend (`POST /auth/register`);
- получает `accessToken` и профиль;
- сохраняет токен и авторизует пользователя.

## Что такое токен

Токен — это “ключ доступа”.
Если он есть и валиден, backend считает, что пользователь вошел.

## Где хранится токен

- в `localStorage` под ключом `vito-access-token`.

## Как frontend понимает, что пользователь авторизован

- есть `AuthContext` (`AuthProvider`);
- если есть токен, frontend делает запрос `GET /user/profile`;
- если профиль успешно пришел — пользователь считается авторизованным.

## Что происходит после обновления страницы

- frontend читает токен из `localStorage`;
- пробует загрузить профиль;
- если профиль не загрузился (например, токен умер), токен очищается.

## Как работает logout

- токен удаляется из `localStorage`;
- текущий пользователь в памяти сбрасывается;
- auth/ad query-кэш очищается.

## Как защищаются private routes

- используется `RequireAuth`;
- если пользователь не авторизован — редирект на `/login`;
- если авторизация еще проверяется, показывается текст “Проверяем авторизацию...”.

## Зачем нужен `isAuthChecked`

Это флаг “проверка авторизации завершена”.
Он нужен, чтобы не делать ложный редирект на `/login` до того, как frontend успеет проверить токен и профиль.

# 6. Как frontend работает с объявлениями

## Где список объявлений

- общий список: `/ads` (режим `all`);
- личный список: `/my-ads` (режим `my`).

Обе страницы используют один общий каталог-компонент с разным режимом.

## Где детали объявления

- публично: `/ads/:id`;
- в личном разделе: `/my-ads/:id`.

## Где создание нового объявления

- `/my-ads/create`, только для авторизованных.

## Где редактирование

- `/ads/:id/edit` и `/my-ads/:id/edit`.

## Кто может редактировать

- только авторизованный владелец объявления.
- если объявление чужое, в edit-странице показывается предупреждение “Нет доступа...”.

## Кто может создавать

- только авторизованный пользователь.

## Кто видит owner-only действия

На деталке:
- кнопка “Редактировать” — только если пользователь владелец;
- кнопка “Удалить” — только если владелец и открыт путь `/my-ads/...`.

В edit:
- delete-кнопка есть в личном разделе (`/my-ads/.../edit`);
- на публичном edit-пути (`/ads/:id/edit`) delete-кнопка не показывается.

## Как это примерно устроено в коде

- ownership проверяет helper `canManageAd(...)`;
- он смотрит `isMine`, `ownerId`, а также fallback-поля;
- есть временный route-fallback для `/my-ads` (если backend не дал owner-мета).

# 7. Как frontend работает с backend API

## Base URL

Во frontend настроен:

- `http://localhost:8080`

## Какие endpoints реально вызываются

### Auth
- `POST /auth/login`
- `POST /auth/register`
- `GET /user/profile`

### Объявления
- `GET /items`
- `GET /items/:id`
- `PUT /items/:id`
- `POST /ads`
- `DELETE /ads/:id`

### AI
- `POST /ai/description`
- `POST /ai/price`
- `POST /ai/analyze`

## Что важно помнить про API

- токен подставляется автоматически через axios interceptor в `Authorization: Bearer ...`;
- список/детали/редактирование идут через `/items`;
- создание/удаление идут через `/ads`.

## Важная особенность (может путать)

Да, сейчас так и есть:
- чтение + update = `/items`;
- create + delete = `/ads`.

Это рабочая схема, но для нового человека выглядит немного странно.

# 8. AI-функции

## Есть ли AI-помощники

Да.

## Что они делают

На create/edit доступны три функции:
- улучшить/сгенерировать описание;
- подсказать ориентир по цене;
- сделать структурированный анализ объявления.

## Нужен ли Ollama

Да, но не напрямую во frontend.

Сейчас логика такая:
- frontend вызывает backend (`/ai/*`);
- backend уже вызывает Ollama.

## Что уже реально работает

- кнопки AI и состояния loading/success/error;
- вывод результатов;
- применение сгенерированного описания в форму;
- вывод структуры анализа (summary/strengths/weaknesses/recommendations).

## Что еще не видно во frontend

- отдельного индикатора “AI сейчас доступен / недоступен” нет;
- пользователь узнает о проблеме AI по ошибке запроса в конкретном блоке.

# 9. Ограничения текущего frontend

Ниже только то, что видно по текущему коду.

## Чего сейчас нет / что ограничено

- Нет отдельной frontend-страницы “Профиль пользователя” с редактированием личных данных.
- Для create-формы нет draft-восстановления из `localStorage` (draft есть для edit, но не для create).
- Нет выделенного глобального экрана “нет доступа” с редиректом — в edit показывается локальный `Alert`.

## Где frontend зависит от backend

- почти весь контент зависит от backend;
- если backend не запущен, страницы каталога/деталей/auth/create/edit не работают;
- AI зависит от backend `/ai/*` и дальше от Ollama на стороне backend.

## Какие вещи временные / спорные

- helper `canManageAd` содержит fallback `assumeOwnOnMyAdsRoute` (временная страховка на случай неполных owner-данных от backend).
- API разделен на `/items` и `/ads` для одной и той же сущности объявления.

## Где могут быть сюрпризы для нового человека

- удаление объявления доступно из UI только в сценариях личного раздела;
- create после успеха ведет на публичную деталку `/ads/:id`, а не на `/my-ads/:id`;
- кнопка “Войти через Госуслуги” пока заглушка (disabled), без интеграции.

# 10. Что стоит улучшить дальше

Коротко и по делу:

- сделать единый стиль API-роутов (`/items` vs `/ads`);
- добавить draft для create-формы (как в edit);
- убрать временный ownership-fallback после стабилизации backend-поля `ownerId/isMine`;
- добавить явный индикатор доступности AI (например, `/ai/status`);
- добавить более понятный UX для “нет доступа” (например, аккуратный редирект на деталку).

# 11. Короткая памятка для нового человека

Если ты только открыл этот frontend, запомни вот это:

1. Главная рабочая страница — `/ads`, личный раздел — `/my-ads`.
2. Приватные роуты закрыты через `RequireAuth`.
3. Токен хранится в `localStorage` (`vito-access-token`).
4. После перезагрузки сессия восстанавливается через `GET /user/profile`.
5. Читать/редактировать объявления frontend ходит в `/items`, а создавать/удалять — в `/ads`.
6. AI-кнопки работают через backend `/ai/*`, не напрямую к Ollama.
7. Если что-то “не пускает”, сначала проверь: есть ли токен, корректен ли owner, и запущен ли backend.
