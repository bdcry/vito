# Локальные заметки по FSD

## Слои

- `app` — точка входа в приложение: провайдеры, роутер, глобальные стили.
- `pages` — страницы по роутам.
- `widgets` — крупные блоки страницы.
- `features` — действия пользователя.
- `entities` — основные бизнес-сущности.
- `shared` — переиспользуемые общие утилиты и UI.

## Направление зависимостей

- `app` может использовать всё, что ниже.
- `pages` могут использовать `widgets`, `features`, `entities`, `shared`.
- `widgets` могут использовать `features`, `entities`, `shared`.
- `features` могут использовать `entities`, `shared`.
- `entities` могут использовать только `shared`.
- `shared` не должен зависеть от верхних слоёв.

## Соответствие для VITO

- `entities/ad`
  - `api` — `getAds`, `getAdById`, `updateAd`
  - `model` — типы `Ad`, мапперы, нормализация
  - `lib` — хелперы для цены, даты, категории
  - `ui` — `AdCard`, `RevisionBadge` и мелкий UI, связанный с объявлением

- `features`
  - `search-ads` — поиск и обновление `q`
  - `filter-ads` — фильтры по категории и `needsRevision`
  - `sort-ads` — сортировка
  - `paginate-ads` — пагинация
  - `save-ad-draft` — сохранение черновика в `localStorage`
  - `restore-ad-draft` — восстановление черновика
  - `generate-description` — AI-запрос на описание
  - `suggest-price` — AI-запрос на цену

- `widgets`
  - `ads-toolbar` — поиск, сортировка, общее количество
  - `ads-filters` — блок фильтров
  - `ads-list` — список карточек объявлений
  - `ad-details` — блок деталей объявления
  - `ad-edit-form` — блок формы редактирования
  - `ai-assistant-panel` — AI-кнопки и результаты

- `pages`
  - `ads-list-page` — `/ads`
  - `ad-details-page` — `/ads/:id`
  - `ad-edit-page` — `/ads/:id/edit`

## Простые правила

- Если это целый экран, клади в `pages`.
- Если это большой визуальный блок, клади в `widgets`.
- Если это действие пользователя, клади в `features`.
- Если это основной объект приложения, клади в `entities`.
- Если это можно переиспользовать где угодно и оно не знает ничего про объявления, клади в `shared`.

## Поток данных для `/ads`

1. `router` открывает `ads-list-page`.
2. `ads-list-page` читает search params или локальное состояние страницы.
3. `ads-list-page` вызывает `useQuery`.
4. Функция запроса использует `entities/ad/api/getAds`.
5. `ads-toolbar` и `ads-filters` меняют query params.
6. Query params меняют `queryKey`.
7. TanStack Query заново запрашивает список.
8. `ads-list` получает готовый массив `items`.
9. `ads-list` рендерит `entities/ad/ui/AdCard`.

## Важное замечание

- `AdCard` не должен сам запрашивать весь список.
- Страница или крупный виджет должны получить список один раз и передать данные вниз через `props`.
- Фильтры влияют на параметры запроса, а не на сам компонент карточки.

## Принятое упрощение по `pages`

- Не раскладывать страницы по полной FSD-декомпозиции, если это не даёт пользы под дедлайн.
- Для каждой страницы внутри `pages/<page-name>` можно держать простую структуру:
  - `ui` — основной файл страницы
  - `components` — локальные подкомпоненты этой конкретной страницы
- Если логика и UI нужны только одной странице, их можно оставлять внутри её папки, а не выносить в `widgets/features` ради формальности.
- Это решение нужно будет коротко описать в финальном `README` как осознанное упрощение под сроки.
