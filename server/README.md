# 1. Что это за backend

Это сервер для проекта **VITO** (упрощенная копия Авито).

Простыми словами:
- frontend (интерфейс в браузере) показывает страницы;
- backend (этот сервер) хранит и отдает данные;
- backend проверяет права (кто может редактировать/удалять);
- backend делает авторизацию (вход/регистрация);
- backend ходит в Ollama для AI-подсказок.

Сервер работает с frontend VITO по HTTP API (это набор URL-ручек, куда frontend отправляет запросы).

---

# 2. Что умеет сервер сейчас

Сейчас реализовано:
- регистрация пользователя;
- логин пользователя;
- получение профиля текущего пользователя;
- список объявлений с поиском, фильтрами, сортировкой и пагинацией;
- получение деталей объявления;
- создание объявления;
- редактирование объявления;
- удаление объявления;
- AI-ручки через Ollama:
  - генерация/улучшение описания;
  - подсказка цены;
  - анализ объявления.

Что важно:
- данные хранятся **в памяти процесса** (не в базе данных);
- после перезапуска сервера изменения теряются;
- logout как отдельной backend-ручки сейчас не реализован.

# 4. Как работает авторизация

Коротко и просто:

- **Токен** — это строка, которую сервер выдает после успешного логина/регистрации.
- Токен нужно передавать в заголовке `Authorization`.
- Формат заголовка:

```http
Authorization: Bearer <ваш_токен>
```

- Если не передать токен в защищенную ручку, сервер вернет `401 Unauthorized`.

Как токен создается сейчас:
- сервер делает строку вида `token-<userId>-<uuid>`;
- хранит ее в памяти (Map);
- при перезапуске сервера все токены сбрасываются.

# 5. Полный список API-ручек

Base URL:

`http://localhost:8080`

## POST /auth/register

Что делает:
- создает нового пользователя;
- сразу возвращает токен и профиль.

Токен нужен:
- нет.

Body:

```json
{
  "email": "new-user@vito.ru",
  "password": "123456",
  "fullName": "Новый Пользователь"
}
```

Успешный ответ (`200`):

```json
{
  "accessToken": "token-user-4-4f0f0f0f-aaaa-bbbb-cccc-111122223333",
  "user": {
    "id": "user-4",
    "email": "new-user@vito.ru",
    "fullName": "Новый Пользователь"
  }
}
```

Ошибки:
- `409` — пользователь с таким email уже есть.
- `400` — не прошла валидация (например, короткий пароль или некорректный email).

Что это значит простыми словами:
- `409`: такой аккаунт уже зарегистрирован.
- `400`: данные в запросе не подходят под правила.

## GET /items

Что делает:
- возвращает список объявлений.

Токен нужен:
- обычно нет;
- нужен, если используете `mine=true`.

Query-параметры:
- `q` — поиск по названию;
- `limit` — сколько объявлений вернуть;
- `skip` — сколько пропустить (для пагинации);
- `categories` — категории через запятую (`auto,real_estate`);
- `needsRevision` — только объявления, которым нужна доработка (`true` или `1`);
- `mine` — только объявления текущего пользователя (`true` или `1`), работает только с токеном;
- `sortColumn` — `title` или `createdAt`;
- `sortDirection` — `asc` или `desc`.

Пример запроса:

```http
GET /items?q=bmw&limit=10&skip=0&categories=auto&sortColumn=createdAt&sortDirection=desc
```

Успешный ответ (`200`):

```json
{
  "items": [
    {
      "id": 17,
      "category": "auto",
      "title": "BMW X1",
      "price": 300000,
      "description": "Описание",
      "ownerId": "seed-user-2",
      "isMine": false,
      "needsRevision": false
    }
  ],
  "total": 42
}
```

Ошибки:
- `401` — если `mine=true`, но токен не передан.

Важно:
- `limit` принудительно ограничивается до `10` (даже если передать больше).

# 6. Объявления: как backend их понимает

## Основные поля объявления

- `id` — уникальный id объявления;
- `category` — категория (`auto`, `real_estate`, `electronics`);
- `title` — название;
- `description` — описание;
- `price` — цена;
- `createdAt` / `updatedAt` — даты создания/обновления;
- `params` — дополнительные поля категории;
- `needsRevision` — флаг “нужна доработка”;
- `ownerId` — id владельца;
- `isMine` — это мое объявление или нет (для текущего токена).

## Что такое `category`

Это тип объявления:
- `auto` — авто;
- `real_estate` — недвижимость;
- `electronics` — электроника.

## Что такое `params`

Это специфичные поля для категории.
Пример:
- для `auto`: `brand`, `model`, `yearOfManufacture`, `transmission`, `mileage`, `enginePower`;
- для `real_estate`: `type`, `address`, `area`, `floor`;
- для `electronics`: `type`, `brand`, `model`, `condition`, `color`.

## Что такое `needsRevision`

`needsRevision = true`, если:
- нет описания, или
- параметры категории заполнены не полностью/невалидно.

То есть это сигнал “объявление стоит доработать”.

## Краткий объект в списке vs полный объект в деталке

В `GET /items` (список) приходит сокращенная карточка:
- без `params`;
- без `createdAt/updatedAt`;
- только базовые поля + owner/meta.

В `GET /items/:id` (детали) приходит полный объект:
- с `params`,
- с датами,
- с `seller`.

## Где `/items`, а где `/ads`

Сейчас API смешанное:
- чтение и обновление: `/items` (`GET /items`, `GET /items/:id`, `PUT /items/:id`);
- создание и удаление: `/ads` (`POST /ads`, `DELETE /ads/:id`).

Почему это может путать:
- визуально кажется, что это две разные сущности, хотя по факту это одно и то же объявление.

# 7. Ограничения текущего backend

Ниже только то, что реально видно в коде сейчас.

## Чего сейчас нет

- Нет базы данных (все в памяти процесса).
- Нет refresh token flow (продления токена).
- Нет отдельного logout endpoint.
- Нет endpoint для проверки статуса AI (типа `/ai/status`).

## Что работает не полностью / временно

- Токены живут только пока жив процесс сервера.
- После рестарта сервера все зарегистрированные пользователи и созданные объявления теряются.
- AI-ручки сильно зависят от локальной Ollama (если она не стартует — будет `502`).

## Странности API

- Роуты смешаны: чтение/апдейт через `/items`, создание/удаление через `/ads`.
- Для AI-ручек нет строгой Zod-валидации входного body (в отличие от auth/items).
- В `GET /items` валидация query не обернута в try/catch, поэтому ошибки формата query стоит проверить вручную (как именно они оформляются на ответе).

## Где frontend может упереться

- В limit больше 10: backend все равно режет до 10.
- В AI: если Ollama не установлена/не запустилась, AI-кнопки будут падать по `502`.
- В данных после рестарта: созданные объявления “исчезнут”, потому что хранения на диске/в БД нет.

# 9. Короткая памятка для нового человека

Если ты только открыл этот backend, запомни вот это:

1. Сервер поднимается командой `npm start` и слушает `8080`.
2. Все данные сейчас в памяти, после рестарта все сбрасывается.
3. Защищенные ручки требуют `Authorization: Bearer <token>`.
4. `GET /items` и `GET /items/:id` — чтение, `POST /ads` и `DELETE /ads/:id` — создание/удаление.
5. `needsRevision` показывает, что объявление неполное (обычно нет описания или параметров).
6. AI работает через backend, но зависит от доступности Ollama.
7. Если видишь `401` — проблема в токене; если `502` на AI — проблема в Ollama.
