# 1. Что это за инфраструктура проекта

С точки зрения запуска у VITO сейчас очень простая инфраструктура:

- **frontend** (сайт в браузере);
- **backend** (сервер с API);
- **Ollama** (локальный AI-сервис, который использует backend).

Что должен поднять инженер:

1. backend,
2. frontend,
3. проверить, что Ollama доступна (или дать backend запустить ее автоматически).

Отдельной “большой” инфраструктуры (Docker-кластеры, reverse proxy, CI/CD pipeline, мониторинг, staging/prod окружения) в репозитории сейчас нет.

---

# 2. Какие сервисы есть в проекте

## Frontend (`/frontend`)

Зачем нужен:
- показывает страницы пользователю;
- отправляет запросы в backend.

Как запускается:
- `npm run dev` в папке `frontend`.

От чего зависит:
- от backend по адресу `http://localhost:8080`.

---

## Backend (`/server`)

Зачем нужен:
- отдает API для объявлений, авторизации и AI;
- хранит данные в памяти процесса;
- проверяет права доступа.

Как запускается:
- `npm start` в папке `server`.

От чего зависит:
- от Node.js;
- от Ollama (только для AI-ручек).

---

## База данных

Сейчас отдельной базы данных **нет**.

Как сейчас устроено:
- backend берет начальные данные из `server/data/items.json`;
- дальше хранит объявления и токены в памяти процесса.

Что это значит:
- после перезапуска backend изменения теряются.

---

## AI / Ollama

Зачем нужен:
- генерация/улучшение описания;
- подсказка цены;
- анализ объявления.

Как запускается:
- backend пытается автоматически запустить `ollama serve` (если `OLLAMA_AUTOSTART` не отключен).

От чего зависит:
- от установленного бинарника `ollama`;
- от доступности Ollama API (по умолчанию `http://127.0.0.1:11434`).

---

## CI/CD

В проекте сейчас **не реализовано**:
- нет project-level GitHub Actions;
- нет GitLab CI;
- нет Jenkinsfile;
- нет других pipeline-конфигов в самом репозитории проекта.

---

# 3. Локальный запуск всего проекта

Ниже пошагово, как для первого запуска.

## Что установить заранее

- Node.js (актуальная LTS версия);
- npm;
- Ollama (если хотите AI без ошибок).

## Шаг 1. Запустить backend

```bash
cd server
npm install
npm start
```

Ожидаемо:
- backend слушает порт `8080`;
- backend проверяет Ollama и может попытаться поднять ее сам.

## Шаг 2. Запустить frontend (в новом терминале)

```bash
cd frontend
npm install
npm run dev
```

Ожидаемо:
- frontend поднимется на `http://localhost:5173`.

## Как проверить, что всё поднялось

- открыть `http://localhost:5173` — должна открыться страница `/ads`;
- проверить, что список объявлений загружается;
- в backend-логах нет постоянных ошибок;
- AI-кнопки работают (или дают понятную ошибку, если Ollama недоступна).

## Как понять, что один сервис не видит другой

- frontend не видит backend:
  - в браузере ошибки сети к `http://localhost:8080`;
  - страницы с данными пустые/с ошибкой.
- backend не видит Ollama:
  - AI-запросы возвращают `502`;
  - в логах backend предупреждения про AI/Ollama.

# 4. Переменные окружения

В коде backend реально используются такие переменные:

- `PORT`  
  Где читается: `server/server.ts`  
  Значение по умолчанию: `8080`  
  Обязательна: нет  
  Если не задать: backend поднимется на `8080`.

- `OLLAMA_BASE_URL`  
  Где читается: `server/server.ts`  
  Значение по умолчанию: `http://127.0.0.1:11434`  
  Обязательна: нет  
  Если не задать: backend использует локальную Ollama по умолчанию.

- `OLLAMA_MODEL`  
  Где читается: `server/server.ts`  
  Значение по умолчанию: `qwen3:4b`  
  Обязательна: нет  
  Если не задать: используется `qwen3:4b`.

- `OLLAMA_AUTOSTART`  
  Где читается: `server/server.ts`  
  Значение по умолчанию: `true` (если не равно строке `"false"`)  
  Обязательна: нет  
  Если не задать: backend попробует автозапустить Ollama.

- `OLLAMA_AUTO_PULL`  
  Где читается: `server/server.ts`  
  Значение по умолчанию: `true` (если не равно строке `"false"`)  
  Обязательна: нет  
  Если не задать: backend попробует подтянуть модель.

По frontend:
- `import.meta.env` в коде сейчас не используется;
- base URL backend захардкожен в `frontend/src/shared/api/axios.ts` (`http://localhost:8080`).

Файлы `.env*`:
- в репозитории сейчас нет project-level `.env` и `.env.example`.

# 6. CI/CD

Сейчас CI/CD в проекте **нет**.

Что есть вместо этого:
- локальные npm-скрипты:
  - frontend: `dev`, `build`, `lint`, `lint:fix`, `format`, `format:check`, `preview`;
  - backend: `start`.

Что это значит practically:
- проверки и сборка делаются вручную локально;
- автозапуска тестов/линта по PR сейчас нет;
- автоматического деплоя сейчас нет.

# 8. Как устроен путь запроса через систему

Простая цепочка:

1. Пользователь открывает frontend (`localhost:5173`).
2. Frontend отправляет API-запрос в backend (`localhost:8080`).
3. Backend читает/меняет данные в памяти процесса.
4. Если это AI-запрос:
   - backend отправляет запрос в Ollama (`127.0.0.1:11434`);
   - получает ответ;
   - возвращает его frontend.

Важно:
- frontend напрямую Ollama не вызывает.

# 9. Типичные проблемы при запуске

Ниже реальные и вероятные проблемы для этого проекта.

## Backend не поднялся

Что проверить:
- вы в папке `server`;
- `npm install` выполнен;
- порт `8080` не занят другим процессом;
- версия Node.js достаточно новая.

## Frontend не видит backend

Симптом:
- в UI ошибки загрузки;
- в браузере запросы к `http://localhost:8080` падают.

Что проверить:
- backend действительно запущен;
- frontend запущен отдельно;
- backend слушает правильный порт.

## Ollama не отвечает

Симптом:
- AI-функции дают ошибку;
- backend возвращает `502` на `/ai/*`.

Что проверить:
- установлен ли `ollama`;
- запущен ли `ollama serve` (если автозапуск не сработал);
- правильный ли `OLLAMA_BASE_URL`.

## Порт занят

Симптом:
- сервис не стартует и ругается на `EADDRINUSE`.

Что делать:
- освободить порт;
- или поменять `PORT` для backend;
- для frontend можно запустить Vite на другом порту (через параметры запуска).

## env не подхватились

Что проверить:
- переменные действительно экспортированы в shell до запуска backend;
- нет опечаток в именах (`OLLAMA_AUTOSTART`, `OLLAMA_AUTO_PULL`, и т.д.).

## CORS

В текущем backend CORS уже открыт (`Access-Control-Allow-Origin: *`), поэтому типовая CORS-проблема для локалки маловероятна.

## Build / lint

- frontend: есть `build` и `lint` scripts;
- backend: отдельного `build`/`lint` script сейчас нет.

## Token issues (инфраструктурный аспект)

Токены живут в памяти backend.
После перезапуска backend ранее выданные токены перестают работать — это ожидаемо для текущей реализации.

# 11. Что стоит сделать DevOps следующим шагом

Приоритетные шаги (самая практичная польза):

1. Добавить `docker-compose` для frontend + backend + Ollama.
2. Вынести frontend `API base URL` в env (чтобы убрать хардкод `localhost:8080`).
3. Добавить CI pipeline минимум с `frontend lint/build` и `backend type/lint` проверками.
4. Добавить health endpoint на backend (например, `/health` и `/ai/status`).
5. Подготовить production/staging конфиги (env-файлы/секреты/процессы запуска).
6. Подключить постоянное хранилище (БД) для данных и сессий.

---

# 12. Короткая памятка для DevOps

Если ты отвечаешь за инфраструктуру VITO, сначала проверь вот это:

1. Backend и frontend запускаются отдельно (две папки, два процесса).
2. Backend по умолчанию на `8080`, frontend обычно на `5173`.
3. Frontend жестко ходит в `http://localhost:8080`.
4. AI идет через backend `/ai/*`, а не напрямую из браузера.
5. Для AI нужна доступная Ollama (`127.0.0.1:11434` по умолчанию).
6. Данные и токены backend пока in-memory (не переживают рестарт).
7. Docker/CI/CD/monitoring сейчас нет — это зона для ближайших улучшений.
8. CORS в backend сейчас открыт (`*`) и в локальной разработке обычно не мешает.
9. Если AI падает, смотри backend-логи и доступность `ollama`.
10. Если auth “ломается после рестарта”, это ожидаемо при текущем хранении токенов.
